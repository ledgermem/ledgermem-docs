---
title: Security
description: How LedgerMem protects your data, and what you should do on top.
---

LedgerMem stores potentially sensitive information about your users — facts
they have shared, conversations they have had, decisions they have made.
Security is not bolted on; it is the boundary the rest of the product is
built around.

## Data isolation

- Every memory belongs to exactly one workspace, enforced at the database
  layer with row-level security in PostgreSQL.
- API keys are workspace-scoped — a leaked key can never read another
  workspace's data.
- Workspaces are pinned to a region; data never replicates across
  regions unless you opt in.
- For self-hosted deployments, isolate workspaces further with separate
  database schemas if you want a defense-in-depth posture.

## Encryption

- **In transit:** TLS 1.3 only. The HSTS header is set with a 1-year
  max-age and `includeSubDomains`. We block TLS 1.2 outright.
- **At rest:** AWS KMS for the managed offering; in self-hosted, we
  honor whatever encryption your storage layer provides (e.g.
  PostgreSQL with `pg_crypto` or LUKS-backed volumes).
- **Embeddings:** stored alongside content with the same encryption.
- **Key rotation:** managed deployment rotates the per-workspace
  encryption key annually, automatically.

## Authentication

- API keys are bcrypt-hashed in storage; we never log the key value.
- Keys can be scoped to roles (`read`, `write`, `admin`).
- Optional IP allowlist per key.
- Optional TTL per key (e.g. for the
  [playground](https://play.proofly.dev) which mints 1-hour temp keys).

## Audit logging

Every state-changing API call is recorded in an audit log with:

- The API key ID (not the key itself).
- The acting user (when the call comes through the dashboard).
- The endpoint, request ID, and timestamp.
- The before/after diff for `PATCH` calls.

Logs are retained for 90 days in the managed offering; self-hosted
deployments can configure retention freely.

## Secrets handling

- API keys, OAuth tokens, and webhook secrets are stored in the same
  encrypted store and never appear in logs.
- The dashboard never shows a key value after creation — copy it once,
  store it in your secret manager.
- We support Doppler, AWS Secrets Manager, GCP Secret Manager, and
  Vault for self-hosted deployments via the `secrets` Helm value.

## Rate limiting

- Per-workspace QPS limits prevent toll fraud and runaway clients from
  taking down the API.
- Limits scale automatically with your plan.
- The `X-RateLimit-*` headers tell every client where it stands.

## What you should do

1. **Use one workspace per tenant** if you build a multi-tenant SaaS.
2. **Rotate API keys** at least annually, more often for sensitive
   environments.
3. **Use scoped keys** — read-only for your read path, write keys only
   where they are needed.
4. **Enable IP allowlists** on production keys.
5. **Mask PII at write time.** LedgerMem does not auto-redact; if you
   need PII filtering, run your own redactor before calling
   `POST /v1/memories`.
6. **Audit periodically.** Pull the audit log and look for keys that
   have not been used in 30+ days — rotate or delete them.

## Compliance

- **SOC 2 Type II:** report available under NDA from
  trust@proofly.dev.
- **GDPR:** EU region available; data subject erasure honored within
  72 hours of a verified request.
- **HIPAA:** BAA available on the Enterprise plan; covered entities
  should use the dedicated `hipaa-*` regions.
- **CCPA:** delete-on-request honored within 45 days.

## Reporting a vulnerability

Please email security@proofly.dev with as much detail as you can provide.
We respond within 48 hours and aim to remediate critical issues within 14
days. We run a public bug bounty on HackerOne — see the
[security page](https://proofly.dev/security) for scope and rewards.
