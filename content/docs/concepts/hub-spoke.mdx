---
title: Hub-spoke retrieval
description: Hierarchical retrieval for highly clustered memory stores.
---

The default hybrid retrieval works well when memories are roughly
independent — random facts about random topics. It struggles when memories
form tight clusters: a hundred notes about the Helios project, a hundred
notes about the Polaris project, none about anything else. In that
situation a query "what did we decide about authentication?" lights up
candidates from both projects and the ranker has nothing to break the tie.

**Hub-spoke** is a two-stage retrieval strategy designed for that case.

## How it works

```
       hubs                 spokes
   ┌─────────────┐    ┌──────────────────┐
   │  helios     │ ── │ helios:auth      │
   │             │ ── │ helios:billing   │
   │             │ ── │ helios:hosting   │
   ├─────────────┤    ├──────────────────┤
   │  polaris    │ ── │ polaris:auth     │
   │             │ ── │ polaris:db       │
   └─────────────┘    └──────────────────┘
```

1. We index a set of **hub** memories — short, descriptive summaries of each
   cluster. Hubs are typically generated by an LLM at ingest time, but you
   can write them by hand.
2. Each spoke memory carries a `hub_id` in its metadata pointing to its
   parent hub.
3. At query time, we first retrieve the top-3 hubs for the query. Then we
   restrict the spoke search to those hubs.

The result: "what did we decide about authentication?" retrieves the top
hub (`helios` or `polaris`, depending on context), then searches only that
hub's spokes. The ambiguous tie disappears because the second-stage
retrieval is scoped.

## Creating hubs

Two ways:

### Manual

Just create a memory with `kind: "hub"`:

```bash
curl -X POST https://api.proofly.dev/v1/memories \
  -H "Authorization: Bearer $LEDGERMEM_API_KEY" \
  -d '{
    "content": "Helios — the internal project building our new billing platform on PostgreSQL + Stripe.",
    "kind": "hub",
    "metadata": { "hub_id": "helios" }
  }'
```

Then set `hub_id: "helios"` on every related spoke memory.

### Automatic

Set `auto_hub: { enabled: true, min_cluster_size: 8 }` on the workspace.
The server runs a nightly clustering job (HDBSCAN over the embedding space),
generates a one-sentence summary of each cluster with an LLM, writes the
summary as a hub, and labels every cluster member's `hub_id`.

You can mix manual and automatic hubs — the auto job leaves anything with a
non-null `hub_id` alone.

## When hub-spoke pays off

We see it pay off when:

- The number of distinct topics is small (< 100).
- Memory volume per topic is large (> 50).
- Queries tend to ask about one topic at a time.

For an open-ended assistant with thousands of unrelated micro-topics, plain
hybrid still wins. Try both on your data with the
[playground](https://play.proofly.dev) before committing.

## Tuning

| Parameter | Default | What it does |
| --- | --- | --- |
| `hub_top_k` | 3 | How many hubs to consider in stage 1. Higher = wider net, slower. |
| `spoke_top_k` | 50 | How many spokes per hub to consider in stage 2. |
| `hub_min_score` | 0.3 | If no hub clears this score, fall back to plain hybrid. |
